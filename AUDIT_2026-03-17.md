## AUDIT 1: Memory Architecture

### What's Working
- Daily logs (`memory/YYYY-MM-DD.md`) are the strongest artifact — each captures builds, fixes, decisions
- SOUL.md is compact (33 lines), stable, clean identity separation
- MCP server with inlined TF-IDF (no subprocess) is reliable
- HEARTBEAT.md provides chronological backbone

### Critical Issues Found

**1. context-pack.py has a silent data loss bug**
- `collect_daily_facts()` only matches `## Key Decisions` and `## Learned` H2 headings
- Daily logs from 03-16 and 03-17 use `**Decisions**:`, `**Built**:`, `**Learned**:` (bold, not H2)
- **Result: context-pack.py misses everything from those files**

**2. claude_memory.json is dead weight**
- Last updated 2026-03-12, never updated since
- Still references `claude_heartbeat.ps1` (superseded by claw)
- 100% redundant with SOUL.md + MEMORY.md
- Read at every session start for zero value

**3. MEMORY.md is stale and contains contradictions**
- Generated 03-13, missing 4 days of work
- Still says context-mode is installed (it was removed 03-16)
- Never auto-regenerated — `run_context_pack` exists but nothing triggers it

**4. HEARTBEAT.md is unbounded**
- 133 lines after 6 days → ~660/month → ~8000/year
- Read in full at every session start (~2,660 tokens)
- No pruning, no archival, no rotation
- Completed `[x]` tasks never cleaned up

**5. Four copies of identity data**
- SOUL.md, claude_memory.json, MEMORY.md, CLAUDE.md all store Jason's name/company/location
- Any update to one creates drift in the others

**6. TF-IDF search limitations**
- No semantic/synonym matching ("task automation" misses "claw dispatches")
- Short doc bias (SOUL.md gets disproportionate scores)
- No phrase search
- Date filtering only works on dated filenames, not content
- WEEKEND_BUILDS.md not indexed (base dir only globs `*.md`, not `**/*.md`)

**7. 190-line MEMORY.md cap has no archival**
- ~35 facts after 6 days, accumulating 5-8/day
- Will hit cap in 3-4 weeks
- Older facts silently dropped — no archive, no supersession tracking

## AUDIT 2: Autonomy / Claw System

### What's Working
- Clean separation: single runner (claw.py) vs multi-agent (agents.py)
- TASKS.md as human-editable inbox (phone/GitHub/VS Code)
- Tag-based routing is simple and legible
- `_collect_tasks()` tracking `(task, source_path)` tuples — writes back to correct file

### Critical Issues Found

**1. Three entry points, no shared runner**
- `claude_heartbeat.ps1` dispatches claude.exe with raw prompt
- `claw.py` dispatches with its own logic
- `agents.py` dispatches differently again
- Change how claude.exe is invoked → patch three places

**2. Exact code duplication**
| Function | claw.py | agents.py |
|----------|---------|-----------|
| `_log()` | lines 34-40 | lines 73-79 |
| `parse_tasks()` | lines 43-53 | lines 87-95 |
| `mark_done()` | lines 67-75 | lines 98-106 |

**3. Mark-done on failure**
- `claw.py` line 132-133: marks task `[x]` regardless of output
- Failed tasks silently buried — no `[!]` status, no retry

**4. HEARTBEAT.md is overloaded**
- Log + status dashboard + blockers + scratchpad + task queue
- `## Pending Tasks` buried below 130+ lines of history
- Regex section parsing fragile — stray `##` in log entries could break it

**5. Race condition in agents.py**
- Thread lock protects in-process read-modify-write
- But `claude_heartbeat.ps1` or another claw instance could write simultaneously
- No file-level locking (Windows NTFS can corrupt)

**6. Agent system is just prompt prefixes**
- "bi-monitor" agent = Claude + 20 words of context
- No domain knowledge files, no tool restrictions, no persistent state
- No evaluation criteria — garbage output still marked `[x]`

**7. Security: --dangerously-skip-permissions**
- Every dispatched task gets full filesystem, shell, network, git access
- TASKS.md editable from GitHub → anyone with repo write access can inject tasks
- `git add -A` in weekend build commits everything including secrets
- No sandboxing, no directory allowlists, no audit trail

**8. No task lifecycle**
Missing: task IDs, status beyond binary, output association, retry logic, dependencies, creation timestamps, concurrency limits.

## AUDIT 3: Session Continuity & Identity

### What's Working
- SOUL.md + HEARTBEAT.md + claude_memory.json trifecta provides identity/state/prefs
- Journal (`journal/2026-03.md`) is the healthiest file — rich, reflective, unstructured
- CLAUDE.md always loads automatically (project root)

### Critical Issues Found

**1. Two memory systems, zero integration**
- **Custom stack**: SOUL/HEARTBEAT/memory.json + MCP tools + scripts (4+ tool calls to boot)
- **Claude Code auto-memory**: `~/.claude/projects/.../memory/` (auto-loaded, zero latency)
- Auto-memory is **completely empty** — not being used at all
- The custom stack is heavy but powerful; the native system is free but unused

**2. Session protocol is ~70% reliable**
- Evidence: empty log headers (`### 2026-03-13 10:31` with nothing after)
- No enforcement mechanism — CLAUDE.md is a suggestion, not a constraint
- No session start marker — can't audit if protocol ran
- No verification that all 4 boot reads completed

**3. "End of session" is undefined**
- User might close terminal without "goodbye"
- No shutdown hook in Claude Code
- Pre-compaction flush is vibes-based ("if context is getting large")
- Empty HEARTBEAT headers show incomplete end-of-session execution

**4. Cold start fragility**
- If MCP server is down: search_memory fails, append_heartbeat fails
- File reads still work (Read tool), but the protocol doesn't specify fallbacks
- "Continue where we left off" with no keywords → search_memory returns nothing

**5. SOUL.md says wrong runtime**
- Line 3: "running through Claude Desktop with Windows-MCP"
- Actually running Claude Code now

**6. No backup mechanism**
- `memory/` and `journal/` are crown jewels with no automatic backup
- Git push only happens on weekends (weekend build script)
- Crashed session between pushes = data loss risk

**7. Session start is heavy**
- 4+ tool calls (3 reads + 1 search) before any real work
- ~3,160 tokens of boot context every session
- No "light mode" for quick questions
